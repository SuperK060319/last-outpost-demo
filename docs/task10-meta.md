# 任务 10：战后勋章消费与复盘引导

## 模块设计

局外成长分成两类，并统一由 `src/game/meta/doctrines.js` 管理：

- **战斗加成**：后勤储备每级增加 2 点开局补给；加固工事每级增加 4% 闸门初始与最大耐久。两者最多 3 级，满级单项加成都低于 15%。
- **信息加成**：战报分析不影响战斗，1–3 级依次显示武器位、三线强化分布和防线等级。

`GameStore` 是唯一购买入口。它负责标准化存档、检查等级与余额、扣除勋章并保存。旧存档缺少 `doctrines` 时会自动补为三个 0 级，因此无需迁移步骤。结果页只调用购买入口，不自行修改余额。

战后复盘使用纯函数 `buildBattleReview()`。它根据胜负、抵达波次、两个武器位、强化分布和闸门等级，在结果页给出一条明确建议；分析等级只增加事实详情，不堆叠多条互相竞争的建议。

## 验收标准

1. 旧版 `{ bestWave, jade, wins }` 存档可直接进入游戏。
2. 条令按当前等级价格扣费；余额不足、满级和未知条令均不扣费。
3. 后勤满级为 `+6` 开局补给，加固满级为 `+12%` 闸门耐久。
4. 战报分析不会改变开局补给或闸门耐久。
5. 结果页能购买三项条令、即时刷新余额与详情；菜单页能看到总览。
6. 固定内容全部落在 `540×960` 安全区内，并由 Phaser `FIT` 等比缩放到手机或桌面。

## 调用与安全修改区

- 调整名称、说明、每级价格：修改 `src/game/meta/doctrines.js` 中的 `DOCTRINES`。
- 调整永久加成：修改同文件的 `doctrineBonuses()`；战斗加成建议继续保持单项满级低于 15%。
- 调整复盘优先级或文字：修改同文件的 `buildBattleReview()`。
- 调整结果页行距和位置：修改 `src/scenes/ResultScene.js`，底部交互内容不要超过 `y=890`。
- 不要从场景代码直接改 `profile.jade` 或 `profile.doctrines`，购买必须走 `gameStore.purchaseDoctrine()`，否则会绕过校验与存档。

## 验证方式

```powershell
node scripts/meta-progression-sim.mjs
npm run build
```

模拟脚本覆盖旧存档、正常消费、余额不足、满级、开局加成、房间奖励叠加、信息条令零战力和详情逐级解锁。
